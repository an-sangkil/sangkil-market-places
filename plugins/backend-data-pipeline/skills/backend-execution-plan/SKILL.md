기획 문서를 백엔드 개발자가 바로 착수할 수 있는 실행계획 문서로 변환한다.

## 입력 수집

요청에서 아래 항목을 우선 파악한다. 일부 누락되어도 중단하지 말고 합리적 가정을 명시하며 진행한다.
- PRD / 기능 요구사항 / 비기능 요구사항
- 화면정의서 / 이벤트 플로우
- 외부 연동 문서 / 일정 제약 / 운영 정책

## 요구사항 분석 체크리스트

시작 전 아래 항목을 점검한다. 체크리스트 전문은 [requirement-analysis-checklist.md](requirement-analysis-checklist.md) 참고.

- 해결하려는 문제를 한 문장으로 설명할 수 있는가
- 사용자 액터와 주요 시나리오가 명확한가
- 입력/출력/예외가 정의되었는가
- 외부 API/시스템 의존성이 식별되었는가

## 실행 절차

1. **목표 재정의**: 문제와 목표를 한 문단으로 재정의
2. **요구사항 분해**: 사용자 시나리오 기준으로 분해
3. **백엔드 변환**: 각 시나리오를 백엔드 기능으로 변환
4. **기술 설계 포인트**: 기능별 기술 설계 포인트 도출
5. **작업 단위 분해**: 구현 가능한 단위로 쪼개고 우선순위 매핑
6. **계획 완성**: 일정, 의존성, 리스크, 검증 계획 추가

### 백엔드 관점 변환 기준

| 관점 | 정의 항목 |
|---|---|
| API | 엔드포인트 후보, 요청/응답 구조, 에러 코드, 멱등성 여부 |
| DB | 엔티티, 키, 인덱스, 트랜잭션 경계, 마이그레이션 |
| 이벤트/비동기 | 발행/구독 이벤트, 재처리 전략, 순서 보장, DLQ 대응 |
| 배치/스케줄링 | 주기, 실행 윈도우, 실패 복구 전략 |
| 보안/권한 | 인증, 인가, 민감정보 처리, 감사 로그 |

### 작업 단위 설계

- Epic → Story → Task 3단 분해
- 각 Task에 산출물, 선행 조건, 완료 정의(DoD), 검증 방법 명시
- Task 크기: 0.5~2일 내 완료 가능한 단위
- 우선순위: `Must`(릴리스 필수) / `Should`(중요하지만 지연 가능) / `Could`(여유 시 포함)

## 최종 출력 형식

[backend-execution-plan-template.md](backend-execution-plan-template.md) 형식으로 작성한다.

## 품질 게이트 (완료 판단 기준)

- [ ] 모든 주요 요구사항이 최소 1개 이상의 Task로 매핑됨
- [ ] API/DB/이벤트/보안 관점 누락 없음
- [ ] 우선순위와 일정이 서로 모순되지 않음
- [ ] 테스트 전략이 핵심 성공 기준을 검증함
- [ ] 리스크 대응 전략과 책임 주체가 명시됨

누락 정보는 "추가 확인 필요"로 표시하되, 임시안도 함께 제시한다.
모호한 요구는 비판 대신 대안 1~2개를 함께 제시한다.
