# docs/

프로젝트 문서 루트 디렉토리.

## 구조

| 폴더 | 용도 | 작성 시점 |
|------|------|----------|
| `design/` | 기능별 요구사항, 개발 플랜 (왜, 어떻게) | `/spec`, `/plan` 커맨드 |
| `exec-plans/` | 단건 실행 기록 | `/build` 단독 실행 |
| `output/` | 결과물 (테스트, 리뷰, QA, 배포, 인수인계) | `/test`, `/review`, `/qa`, `/deploy-check`, `/doc-handoff` |
| `architecture/` | 공통 아키텍처 규칙 (파이프라인 구조, 코드 컨벤션 등) | 프로젝트 초기 |
| `product-specs/` | 제품 스펙 (파싱 규칙, DB 스키마, 비즈니스 로직 등) | 수시 |
| `references/` | 개발 참조 가이드 (테스트 패턴, 보안 체크리스트, 트리아지 절차 등) | 프로젝트 초기 |

## 규칙

- `architecture/` 파일은 모든 기능 문서의 상위 참조 — 기능 문서에 공통 구조를 중복 서술하지 않는다.
- `design/{domain}/` 하위에 기능별 설계 문서를 생성한다.
- `output/{domain}/` 하위에 기능별 결과물을 생성한다.
- `references/` 파일은 에이전트의 **단일 기준 문서** — 에이전트가 코드 작성/리뷰 시 이 문서를 근거로 판단한다.
- 구현으로 동작이 변경되면 같은 작업 사이클에서 관련 문서도 함께 갱신한다.
