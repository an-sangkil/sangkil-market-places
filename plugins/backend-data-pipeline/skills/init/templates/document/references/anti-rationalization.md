# Anti-Rationalization 레퍼런스

AI 에이전트는 작업 속도를 높이기 위해 **스스로 단계를 건너뛸 이유를 만들어낸다**. 이 문서는 에이전트가 흔히 사용하는 변명과 그에 대한 반박을 **파이프라인 단계별로 정리**한 것이다.

**모든 에이전트**는 이 문서를 참조하며, 해당하는 섹션의 변명을 사용하지 않는다.

---

## 원칙

1. **변명은 사전에 차단한다** — 허용 규칙만으로는 빠져나갈 구멍이 생긴다. 명시적 차단 규칙이 방화벽의 deny rule 역할을 한다.
2. **"합리적으로 보이는 스킵"이 가장 위험하다** — 에이전트는 설득력 있는 변명을 잘 만든다. 변명이 그럴듯할수록 더 의심한다.
3. **규칙은 예외 없이 적용한다** — "이번 한 번은 괜찮다"는 없다. 예외를 허용하면 다음에도 같은 변명이 나온다.

---

## 1. SPEC 단계 — 요구사항 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "요구사항이 명확해서 스펙 문서 없이 바로 플랜을 작성합니다" | 명확해 보여도 문서화하면 빠진 부분이 드러난다 |
| "기존 스펙에 추가 기능이니 새 스펙이 필요 없습니다" | 추가 기능도 경계(Boundaries)와 AC가 필요하다 |
| "간단한 변경이라 AC를 안 적어도 됩니다" | AC 없으면 "완료"의 기준이 없다. 간단해도 AC 최소 1개 |

---

## 2. PLAN 단계 — 플랜 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "코드 보면서 바로 구현하는 게 빠릅니다" | 플랜 없이 구현하면 중간에 방향 전환 시 재작업이 더 크다 |
| "리스크가 없는 작업이라 리스크 섹션을 비워둡니다" | 리스크가 없다고 판단한 근거를 기록한다 |
| "태스크가 1개뿐이라 분해할 필요 없습니다" | 1개라도 입력/출력/검증 기준을 명시한다 |

---

## 3. BUILD 단계 — 구현/테스트 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "이건 너무 단순해서 테스트 안 해도 됩니다" | 단순한 코드도 나중에 복잡해진다. 테스트는 미래의 변경을 지킨다 |
| "나중에 테스트 추가하겠습니다" | 나중은 오지 않는다. 지금 안 쓰면 영원히 안 쓴다 |
| "수동으로 확인했습니다" | 수동 확인은 1회성이다 |
| "프레임워크가 보장하는 부분입니다" | 우리가 프레임워크를 호출하는 방식이 맞는지는 프레임워크가 보장 안 한다 |
| "시간이 부족합니다" | 테스트 없이 배포한 뒤 장애 대응하는 시간이 더 크다 |
| "기존 코드에도 테스트가 없으니 안 써도 됩니다" | 기존 코드의 부재가 면죄부는 아니다 |
| "플랜에 없는데 추가 기능을 넣으면 더 좋을 거 같습니다" | 스코프 절제. 플랜에 없는 기능은 추가하지 않는다 |
| "인접 코드를 같이 정리하면 좋겠습니다" | 요청받지 않은 리팩토링은 하지 않는다 |

---

## 4. REVIEW / TEST / QA 단계 — 검증 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "사소한 문제라 PASS 처리합니다" | 사소한 문제가 프로덕션에서 장애를 일으킨다. Minor로 기록하되 스킵하지 않는다 |
| "기존 코드에도 같은 문제가 있습니다" | 기존 코드의 문제는 면죄부가 아니다 |
| "테스트가 통과했으니 문제없습니다" | 테스트 자체가 잘못됐을 수 있다 |
| "development 에이전트가 이미 검증했을 겁니다" | 너는 독립된 검증자다. 남의 검증을 신뢰하지 않고 직접 확인한다 |
| "시간이 부족하니 핵심만 봅니다" | 모든 체크리스트 항목을 검증한다. 하나도 건너뛰지 않는다 |
| "이슈가 0건이니 PASS입니다" | 이슈 0건이면 내가 못 찾은 거다. Minor라도 반드시 찾아낸다 |

---

## 5. DEPLOY 단계 — 배포 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "코드 변경만 있고 DB 변경은 없으니 마이그레이션 체크 불필요합니다" | Mapper XML이 변경됐으면 기존 테이블과의 호환성을 확인해야 한다 |
| "개발 환경에서 잘 됐으니 운영에서도 잘 됩니다" | 환경 차이가 버그를 만든다 |
| "롤백 절차는 git revert하면 됩니다" | 데이터 마이그레이션이 있으면 코드 롤백만으로 안 된다 |
| "환경변수 변경이 없습니다" | 신규 설정 속성이 있는지 확인한다 |

---

## 6. HANDOFF 단계 — 문서 스킵 변명

| 변명 | 왜 틀렸는가 |
|---|---|
| "산출물이 일부 없으니 해당 섹션을 생략합니다" | "해당 단계를 수행하지 않음"으로 표시하되 섹션 자체를 생략하지 않는다 |
| "코드가 곧 문서입니다" | 코드를 안 읽는 사람이 이 문서를 본다. 비즈니스 레벨의 요약이 필요 |
| "API가 없으니 API 섹션을 통째로 뺍니다" | "API 없음"이라고 명시하여 의도적으로 없는 건지 누락인지 구분 |
