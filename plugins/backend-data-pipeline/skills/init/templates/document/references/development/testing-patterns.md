# Testing Patterns 레퍼런스

이 프로젝트의 테스트 작성 원칙과 검증 기준을 정의한다.
`development` 에이전트와 `test-qa` 에이전트가 공통으로 참조하는 **단일 기준 문서**이다.

---

## 1. TDD 사이클 — Red → Green → Refactor

```
1. Red      — 스펙/플랜을 보고 테스트를 먼저 작성한다. 실행하면 실패해야 한다.
2. Green    — 테스트를 통과시키는 최소한의 코드를 작성한다.
3. Refactor — 테스트가 통과하는 상태에서 코드를 정리한다.
```

**왜 테스트를 먼저 쓰는가:**
- 테스트를 나중에 쓰면 에이전트가 자기가 만든 코드에 맞춰 테스트를 쓴다 → 검증이 아니라 추인
- 테스트를 먼저 쓰면 **기대 동작의 실행 가능한 스펙**이 된다

---

## 2. 테스트 대상 vs 비대상

### 반드시 테스트하는 것
- **조건 분기가 있는 비즈니스 로직** — if/switch, 상태 전이, 예외 처리 분기
- **데이터 변환 로직** — 원시 데이터 → 도메인 객체 파싱, 필드 매핑
- **실수하기 쉬운 패턴** — null 처리, 날짜 계산, 1:N 관계 저장 순서

### 테스트하지 않는 것
- **단순 getter/setter**, 생성자, Builder 패턴 — 조건 분기가 없는 코드
- **프레임워크가 보장하는 동작** — Spring DI, MyBatis XML 매핑 자체, Lombok 생성 코드
- **단순 위임 메서드** — 다른 컴포넌트를 호출만 하고 로직이 없는 경우
- **설정/상수** — Properties, Enum 값 나열, Config 클래스

---

## 3. 테스트 작성 규칙

### 구조
- **Given-When-Then** 패턴 사용
- 테스트 메서드명은 한글 허용 (기존 패턴 따름)
- `src/test/java/` 아래 동일한 패키지 경로에 배치
- 테스트 리소스: `src/test/resources/` 에 샘플 데이터 배치
- Mock은 **Mockito** 사용

### 테스트 규모
- 클래스당 **Happy path 1개 + Edge case 2~3개 + 주요 예외 케이스 1~2개**
- 커버리지 100%를 추구하지 않는다 — **70% 이상이면 수확 체감 구간**

---

## 4. DAMP 원칙 — 테스트 코드는 중복을 허용한다

**DRY (프로덕션 코드)** vs **DAMP (테스트 코드)**:

| 원칙 | 뜻 | 적용 |
|---|---|---|
| DRY — Don't Repeat Yourself | 중복 제거 | **프로덕션 코드** |
| DAMP — Descriptive And Meaningful Phrases | 서술적, 의미 있게 | **테스트 코드** |

### 규칙
- 각 테스트 메서드만 읽으면 뭘 검증하는지 **바로 보여야 한다**
- 공통 헬퍼로 중복을 제거하느라 테스트의 의도를 숨기지 않는다
- 테스트가 실패하면 다른 파일을 뒤지지 않고 그 테스트 안에서 원인을 파악할 수 있어야 한다

---

## 5. 상태 기반 vs 상호작용 기반 테스트

### 원칙
- **상태 기반 테스트 우선**
- `verify()`는 **부수효과 검증**(이벤트 발행, 외부 API 호출)에만 사용한다
- 내부 메서드 호출 순서를 검증하지 않는다 — 리팩토링하면 깨진다

### 판단 기준

| 검증하려는 것 | 방식 |
|---|---|
| "결과가 맞나?" | **상태 기반** |
| "외부에 이벤트를 발행했나?" | 상호작용 기반 (부수효과) |
| "외부 API를 호출했나?" | 상호작용 기반 (부수효과) |
| "내부 메서드 호출 순서가 맞나?" | **하지 말 것** |

---

## 6. 한 테스트 = 한 개념

- 하나의 테스트 메서드가 **여러 독립적 개념을 검증하지 않는다**
- 실패 시 어떤 개념이 깨졌는지 **바로 알 수 있어야 한다**

---

## 7. Mock 사용 원칙

### 규칙
- Mock은 **외부 의존성 격리**에만 사용한다 (DB, 외부 API, 메시지 큐)
- 내부 컴포넌트 간 연동은 가능하면 **실제 객체** 사용
- Mock이 **3개 이상** 필요한 테스트는 설계 재검토 신호

### Mock 사용 우선순위
```
1. 실제 구현 (Real implementation)  ← 가능하면 이걸
2. Fake (경량 구현, 예: in-memory DB)
3. Stub (미리 정의된 응답만 반환)
4. Mock (호출 검증까지)              ← 정말 필요할 때만
```

## 8. Anti-Rationalization — 테스트 스킵 금지

에이전트가 테스트를 건너뛰기 위해 만들어낼 수 있는 변명과 그에 대한 반박.

### 테스트 작성 시 (development 에이전트)

| 변명 | 왜 틀렸는가 |
|---|---|
| "이건 너무 단순해서 테스트 안 해도 됩니다" | 단순한 코드도 나중에 복잡해진다. 테스트는 미래의 변경을 지킨다 |
| "나중에 테스트 추가하겠습니다" | 나중은 오지 않는다. 지금 안 쓰면 영원히 안 쓴다 |
| "수동으로 확인했습니다" | 수동 확인은 1회성이다. 내일 코드가 바뀌면 깨지는 걸 알 방법이 없다 |
| "프레임워크가 보장하는 부분입니다" | 우리가 프레임워크를 호출하는 방식이 맞는지는 프레임워크가 보장 안 한다 |
| "시간이 부족합니다" | 테스트 없이 배포한 뒤 장애 대응하는 시간이 더 크다 |
| "테스트 대상 목록에 없는 코드입니다" | 목록은 최소 기준이다. 조건 분기가 있으면 목록에 없어도 테스트한다 |

### QA 검증 시 (test-qa 에이전트)

| 변명 | 왜 틀렸는가 |
|---|---|
| "사소한 문제라 PASS 처리합니다" | 사소한 문제가 프로덕션에서 장애를 일으킨다. Minor로 기록하되 스킵하지 않는다 |
| "기존 코드에도 같은 문제가 있습니다" | 기존 코드의 문제는 면죄부가 아니다. 신규 코드에는 같은 실수를 반복하지 않는다 |
| "테스트가 통과했으니 문제없습니다" | 테스트 자체가 잘못됐을 수 있다. 테스트 통과 ≠ 정상 동작 |
| "development 에이전트가 이미 검증했을 겁니다" | 너는 독립된 검증자다. 남의 검증을 신뢰하지 않고 직접 확인한다 |
| "시간이 부족하니 핵심만 봅니다" | 모든 체크리스트 항목을 검증한다. 하나도 건너뛰지 않는다 |

---

## 9. QA 검증 시 테스트 품질 리뷰 기준

test-qa 에이전트가 작성된 테스트 코드의 품질을 리뷰할 때 사용하는 체크리스트.

| 항목 | 기준 | 위반 시 심각도 |
|---|---|---|
| 테스트 커버리지 | 섹션 2의 "반드시 테스트하는 것"에 해당하는 코드에 테스트가 있는가 | Major |
| DAMP 원칙 준수 | 각 테스트만 읽어 의도가 바로 보이는가 | Minor |
| 상태 기반 테스트 우선 | 결과값 검증 가능한데 `verify()`를 썼는가 | Major |
| 한 테스트 = 한 개념 | 여러 독립 개념이 한 테스트에 섞여 있는가 | Minor |
| Mock 사용 적절성 | Mock이 3개 이상인 테스트가 있는가 | Minor (설계 문제 가능성) |
