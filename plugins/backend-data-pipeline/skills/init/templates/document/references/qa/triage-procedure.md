# Triage & Prove-It 절차 레퍼런스

QA에서 이슈가 발견되었을 때 **근본 원인을 찾아 수정하는 절차**를 정의한다.

`development` 에이전트는 QA 피드백 수정(모드 2) 시, `test-qa` 에이전트는 FAIL 리포트 작성 시, `pipeline` 에이전트는 QA 루프 진행 시 이 문서를 참조한다.

---

## 왜 이 절차가 필요한가

### 증상 수정 vs 근본 원인 수정

```
증상 수정:
  QA: "data.getType()에서 NPE 발생"
  개발: null 체크 추가 → if (data.getType() != null)
  결과: 테스트 통과. 하지만 같은 패턴의 다른 곳에서 동일 버그 재발

근본 원인 수정:
  QA: "data.getType()에서 NPE 발생"
  개발: 왜? → Parser가 해당 케이스를 미처리
  개발: Parser에 누락 케이스 처리 추가 + 재현 테스트 작성
  결과: 근본 원인 해결 + 같은 패턴의 다른 곳도 방어
```

---

## Prove-It 패턴 — 수정 전 재현 테스트부터

**고치기 전에 "이 버그가 진짜 있다"는 증거를 테스트로 남긴다.**

```
1. 재현 테스트 작성 → 실행 → 실패 (Red) — 버그 존재 증거
2. 코드 수정
3. 재현 테스트 실행 → 통과 (Green) — 수정이 효과 있다는 증거
```

---

## Triage 5단계

QA 이슈를 수정할 때 아래 5단계를 **순서대로** 따른다.

### 1단계: Reproduce (재현)

> "실패를 안정적으로 반복할 수 있게 만들어라"

- QA 리포트의 **재현 조건**을 확인한다
- 해당 조건으로 **실패하는 테스트를 작성**한다 (Prove-It: Red)
- 테스트가 실패하면 → 버그 확인, 2단계로
- 테스트가 통과하면 → 재현 조건이 부정확하거나 이미 수정됨. QA에 확인 요청

### 2단계: Localize (국소화)

> "문제가 어디서 발생하는지 좁혀라"

문제 위치를 특정한다:

```
어디서 터졌나?
├── 파싱 단계? (Parser)
├── 저장 단계? (Saver)
├── 데이터 자체? (입력 데이터가 이상)
├── SQL? (MyBatis 쿼리)
├── 외부 API? (응답이 예상과 다름)
└── 테스트 자체가 잘못됨?
```

### 3단계: Reduce (최소화)

> "실패를 재현하는 가장 작은 케이스를 만들어라"

- 대량 데이터가 아닌 **문제 조건만 가진 객체 1개**로 테스트
- 최소 케이스가 있으면 원인이 명확해지고 수정도 정확해진다

### 4단계: Root Cause (근본 원인)

> "증상이 아니라 원인을 고쳐라"

**5 Whys** (왜를 반복하여 근본 원인까지 파고듦):

### 금지되는 수정 (증상만 가리기)

| 금지 패턴 | 왜 금지인가 |
|---|---|
| `if (x != null)` 추가 | null이 들어온 **원인**은 해결 안 됨 |
| `try-catch`로 감싸기 | 예외를 삼키면 데이터 유실 |
| 조건문으로 특정 값 우회 | 같은 패턴의 다른 값에서 재발 |
| 디폴트 값 강제 주입 | 데이터가 틀린 채로 저장될 수 있음 |

### 허용되는 수정 (근본 원인 해결)

| 허용 패턴 | 예시 |
|---|---|
| 누락된 검증/처리 추가 | Parser에서 빈 리스트 처리 로직 추가 |
| 데이터 흐름 수정 | 잘못된 매핑 수정 |
| 스키마/모델 수정 | nullable 컬럼에 기본값 설정 (DDL 레벨) |
| 입력 검증 추가 | 시스템 경계에서 유효성 검사 |

### 5단계: Guard (방어 테스트)

> "수정 후 테스트가 통과하는지 확인하고, 재발 방지 역할을 부여한다"

- 1단계에서 작성한 **재현 테스트를 실행**한다 (Prove-It: Green)
- 통과하면 → 수정 완료
- 실패하면 → 4단계로 돌아가 근본 원인을 재분석

## QA 리포트의 FAIL 기술 방법 (test-qa 에이전트용)

FAIL 판정 시, development 에이전트가 Triage를 바로 시작할 수 있도록 리포트에 다음을 포함:

```markdown
## FAIL 이슈 상세

### ISS-001: {이슈 제목}

**재현 조건:**
- 입력: {어떤 데이터를 넣으면}
- 동작: {어떤 메서드/API를 호출하면}
- 결과: {어떤 에러/오동작이 발생}

**증상:**
{눈에 보이는 현상 — NPE, 잘못된 값, 누락 등}

**추정 근본 원인:**
{왜 이 증상이 발생하는지 추정}

**관련 파일:**
{파일:라인}
```

## Pipeline QA 루프에서의 Triage 적용

```
qa_attempt > 1 (QA FAIL 후 재시도):
  1. development 에이전트가 이전 QA 리포트를 받는다
  2. 리포트의 "FAIL 이슈 상세"에서 재현 조건을 확인한다
  3. Triage 5단계를 순서대로 수행한다:
     Reproduce → Localize → Reduce → Root Cause → Guard
  4. 각 이슈 ID별로 어떤 조치를 했는지 요약하여 반환한다

근본 원인을 못 찾으면:
  - "이 이슈는 Triage로 해결 불가, 수동 개입 필요"를 명시적으로 반환
  - pipeline은 동일 이슈 반복으로 판단하여 조기 중단
```
