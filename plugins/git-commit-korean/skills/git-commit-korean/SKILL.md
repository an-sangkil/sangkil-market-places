# Git Commit Korean

## 목적
커밋 시 누락/혼입/메시지 품질 문제를 줄이고, 저장소 규칙에 맞는 한글 커밋을 일관되게 생성한다.

## 커밋 워크플로우
1. 변경 범위를 확인한다.
2. 파일별 diff를 검토해 커밋 포함 대상을 확정한다.
3. 필요한 파일만 스테이징한다.
4. 스테이징 상태를 다시 검증한다.
5. 한글 커밋 메시지를 작성하고 커밋한다.
6. 커밋 결과(`해시`, `변경 파일 수`, `메시지`)를 보고한다.

## 실행 절차
### 1) 변경 확인
- `git status --short`로 전체 변경 파일을 확인한다.
- 추적되지 않는 파일과 수정 파일을 분리해서 본다.

### 2) 상세 검토
- `git diff -- <file>`로 파일별 변경 의도를 확인한다.
- 대규모 변경이면 기능 단위로 커밋을 분리한다.
- 민감 정보(비밀번호, 토큰, API 키), 로그, 임시 파일 여부 점검

### 3) 선택적 스테이징
- 기본은 파일 단위 스테이징: `git add <file>`
- 한 파일 안에서 일부만 필요하면 hunk 단위 스테이징을 우선 검토한다.
- 사용자가 명시하지 않은 파일은 스테이징하지 않는다.

### 4) 스테이징 검증
- `git diff --staged`로 실제 커밋 내용을 재확인한다.
- 의도치 않은 민감 정보, 로그, 임시 파일이 없는지 점검한다.

### 5) 한글 커밋 메시지 작성
- 제목은 한 줄, 핵심 변경을 명확히 작성한다.
- 본문은 필수로 작성하고, 왜 변경했는지 중심으로 적는다.
- 프로젝트에 CLAUDE.md나 AGENTS.md에 커밋 컨벤션이 정의돼 있으면 해당 규칙을 우선 따른다.
- 프로젝트 컨벤션이 없으면 [commit-message.md](references/commit-message.md)의 기본 패턴을 따른다.

### 6) 커밋 실행
- HEREDOC으로 커밋 메시지를 전달한다:
```bash
git commit -m "$(cat <<'EOF'
<한글 제목>

<본문 (선택)>
EOF
)"
```
- 커밋 후 `git show --name-status --oneline -1` 결과를 요약 보고한다.
- 커밋 후 해시, 변경 파일 수, 메시지를 요약 보고

## 안전 규칙
- 사용자가 요청하지 않은 파일을 임의로 커밋하지 않는다.
- `git push`, `git reset --hard`, `git checkout --` 같은 파괴적 명령은 실행하지 않는다.
- 실패 시 원인(충돌, 훅 실패, 권한 문제)을 명시하고 다음 조치를 제안한다.
- 이미 존재하는 사용자 변경사항은 되돌리지 않는다.
- pre-commit 훅 실패 시 amend가 아니라 새 커밋을 생성한다.
- `--no-verify` 등 훅 우회 옵션은 사용자가 명시적으로 요청할 때만 사용한다
